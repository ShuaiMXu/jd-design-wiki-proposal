01
Step 01 · Foundation

15.0 设计语言 录入 DESIGN.md

起点
Week 1
如何做
把已有的"根"先立住
京东 15.0 的设计语言已经沉淀在 wiki / Figma / 文档里。 第一步不是发明新东西，是有秩序地迁移——把颜色、字号、间距、圆角、动效、图标、语气等基础资产，按结构化模板转成 MD。
怎么做
按模块拆，一个个转
  • foundations/color.md — 色板 + 语义色
  • foundations/typography.md — 字号 + 字重 + 行高
  • foundations/spacing.md — 栅格 + 间距
  • foundations/radius.md — 圆角值域
  • foundations/motion.md — 动效规格
  • foundations/voice.md — 文案语气
用 W3C DTCG 规范存 token。
要求的结果
一份可查的根 MD
  • 6-8 份 foundations/*.md 完整可读
  • 每份带 YAML 头（机器可读 token）
  • 和 Figma 现有库 数值对齐（若不一致，15.0 优先）
  • committed 进 Git，有 Owner
协同
视觉为主，Shaka 牵引
  • 视觉设计团队 — 负责提供 15.0 原始 token
  • 基础组件 owner — 对齐 Figma 库
  • 前端架构 — 确认 DTCG 格式能被代码消费
  • 内容策略 — voice.md 协同
02
Step 02 · Role Templates

各角色 初始化模板

Week 1-2
如何做
给每个角色一张"填空卡"
每个组件 MD 由多个角色共同填写。我们先给每个角色一份 初始化模板（填空卡），让他们知道"要填什么、填到哪、填多深"。 避免每次从零开始、每次写成不同姿势。
怎么做
4 个核心角色 × 4 份模板
  • templates/product.md — 业务目标、KPI、商业载荷
  • templates/research.md — 用户画像、痛点、场景数据
  • templates/visual.md — 视觉规范、变体、Token 引用
  • templates/experience.md — 旅程、规则、禁止用法
每份模板 = 结构化段落 + 字段级说明 + 示例。
要求的结果
可复用的角色填空卡
  • 4 份 templates/*.md + 写法指南
  • 每份都被对应角色 POC 确认（我填得来、填得完、填得下一个）
  • 一份 templates/combined-example.md 把 4 份拼成一份完整组件 MD 示范
协同
4 个角色 POC 共建
  • 产品代表 — 确认 product 模板
  • 用研代表 — 确认 research 模板
  • 视觉代表 — 确认 visual 模板
  • 体验设计代表 — 确认 experience 模板
每个 POC 独立填一次模板 → 给反馈 → 迭代。
03
Step 03 · Domain MDs

场域试点 · 自下而上拆分

Week 2-3
如何做
先选 2-3 个场域试点
不贪多——先选 2-3 个代表性场域（推荐 / 点评 / 直播）。 每个场域 自下而上展开：Token → 组件 → 核心页面 → 细拆变体。 底层对齐，上层自然统一。
怎么做
四级逐层拆
  • ① Token — 本场域用 foundations 的哪些变量
  • ② 组件 — 本场域有哪些组件（如点评图文卡 / 视频卡）
  • ③ 核心页面 — 组件用在哪些主干页面（求真官首页 / 商详评价）
  • ④ 细拆 — 子变体 / 异常态 / 节奏场景
要求的结果
2-3 棵完整场域树
  • 每个试点场域都有 Token / 组件 / 页面 / 细拆 4 层 MD
  • 组件 MD 含 产品 + 用研 + 视觉 + 体验 4 段
  • Semver 版本号 + Owner + Reviewer 列表
  • 重要决策跟随 ADR 留痕
协同
多业务线共建
  • 各业务线组件 owner — 认领组件并填写
  • 4 个角色 POC — Review 自己领域段落
  • Steering Committee — 裁决冲突、定稳定性标签
  • Shaka — 每周 office hour 答疑
04
Step 04 · Tool Sync

Figma + Zero 打通 POC

Week 3
如何做
让设计工具消费 MD
MD 是源、Figma / Zero 是消费端。建立双向同步—— MD 改了工具自动更新；设计师在工具里改了组件，提 PR 到 MD。
Zero = 京东自研、类 Figma 的设计产品，需要专项打通。
怎么做
Token + 组件双轨同步
  • Token 同步：DTCG JSON ↔ Figma Tokens Studio / Zero Token 面板
  • 组件同步：Figma Code Connect + Zero Component 绑定 MD
  • PR 回流：设计师在工具里改了组件，自动生成 PR 草稿
  • 变体一致性：MD 的 variants/ 和工具库自动映射
要求的结果
三方对齐，不再漂移
  • Token 在 MD / Figma / Zero 值完全一致
  • Figma 和 Zero 里的组件能跳转到对应 DESIGN.md
  • MD 里的变体能 embed Figma / Zero 链接
  • 设计师改组件 → 20 秒内生成 PR（不需要学 Git）
协同
跨工具团队深度合作
  • 前端架构 — Token 同步工具链
  • Zero 团队 — Zero API / 插件支持
  • Figma Plugin 开发（内部或外采）
  • 设计师 POC — 使用体验反馈

Step 05 · AI Consumption

AI 工具 自动化消费

Week 4
如何做
压轴一步 · 让 AI 真正用起来
前 4 步是基础建设，这一步是兑现价值。 每份 DESIGN.md 自动生成对应 AI Skill， 让 Figma AI / Cursor / Claude / 自研 agent 都能"读懂京东的设计哲学"。
怎么做
MD → Skill → Agent 消费
  • 自动生成 Skill：从 MD 的 AI 字段生成 SKILL.md
  • 兼容 agentskills.io 标准（45+ 主流 agent 可用）
  • Skill Registry：京东内部统一目录，类 shadcn Registry
  • MCP 服务：下游 AI 工具通过 MCP 订阅 Registry
  • 双列卡 skill（v0.5.2 已开源）作为首批接入样本
要求的结果
AI 生成即合规
  • 每个组件都有对应可被调用的 AI Skill
  • Figma AI / Zero AI 生成新屏时自动符合京东规范
  • Cursor / 内部 agent 写代码时自动用对的 Token
  • 设计师审查变体时AI 辅助做跨场景同物种识别
协同
AI / Agent 团队主导
  • AI / Agent 团队 — 生成工具 + Registry + MCP
  • 前端架构 — CI 集成、skill 构建流水线
  • 各组件 owner — 确认自己组件的 skill 准确性
  • Zero / Figma 团队 — AI 工具接入协议
